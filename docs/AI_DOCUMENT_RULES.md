# AI向け ドキュメント作成ルール

> このファイルは、AIアシスタントがドキュメントやコードコメントを作成・編集する際に従うべきルールを定義します。
> 新しいルールが追加された場合は、該当セクションに追記してください。

---

## 1. オブジェクト名・項目名の表記ルール

ドキュメント（設計書・テストケース）やコードコメントでは、以下の形式で統一すること。

### 1.1 基本形式

| パターン | 形式 | 例 |
|---------|------|-----|
| オブジェクト名のみ | [日本語名（API名）] | [取引先（Account）] |
| 項目名のみ | [日本語名（API名）] | [取引先名（Name）] |
| オブジェクト.項目 | [オブジェクト].[項目] | [取引先（Account）].[取引先名（Name）] |
| レコードの説明 | 修飾語 + [オブジェクト] | 個人の[取引先（Account）] |

### 1.2 適用範囲

- ✅ 設計書（.md）の文章部分
- ✅ テストケース（.tsv）
- ✅ コードコメント（Apex / JavaScript 等）
- ❌ コードブロック内の変数名・API呼び出し（そのまま記述）
- ❌ SOQL クエリ内（そのまま記述）
- ❌ インラインコード（`` ` `` で囲まれた部分）

## 2. Apex テストクラスのコードカバレッジ要件

Salesforce では、本番環境へのデプロイ時に **Org 全体のコードカバレッジが 75% 以上** であることが必須。

### 2.1 要件

- Org 全体のカバレッジが 75% 未満の場合、デプロイがブロックされる
- 各 Apex クラス単体でも、できる限り 75% 以上を目指すこと

### 2.2 テストクラス作成時のチェックリスト

- [ ] 正常系（主処理が通るパス）を必ずカバーしているか
- [ ] 主要な異常系・エラー分岐をカバーしているか
- [ ] `if` / `else` の両分岐がカバーされているか
- [ ] `continue` で早期スキップする条件もカバーされているか

### 2.3 注意事項

- テストクラス自体はカバレッジ計算の対象外
- `@isTest` アノテーションのあるクラス・メソッドは除外される
- `Test.startTest()` / `Test.stopTest()` で囲むことで非同期処理も含めてカバレッジを取得できる

## 3. コミットメッセージルール

AIアシスタントがコミットを作成する際は以下のルールに従うこと。

### 3.1 言語・スタイル

- **英語必須**（日本語不可）
- **Imperative mood**（動詞の原形で始める）
  - ✅ `Add`, `Fix`, `Update`, `Remove`, `Rename`, `Refactor`
  - ❌ `Added`, `Fixed`, `変更`, `追加`

### 3.2 フォーマット例

```
Fix: add id-token write permission to Claude workflow
Update tab label and card title for metadata exporter
Add commit rules to AI_DOCUMENT_RULES.md
```

### 3.3 コミットのタイミング

- **確定した変更のみをコミットする**
- 試行錯誤（エラー対応の試みや実験的変更）は確定するまでコミットしない
- 不要なコミットが発生した場合は `git reset --hard` で破棄する
